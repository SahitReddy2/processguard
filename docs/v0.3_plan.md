# ProcessGuard v0.3 — plan

**Status:** approved 2026-06-07 (pending merge of this doc). Execution
starts on Phase A after merge.
**Supersedes:** the deferred Phase 2 from `docs/v0.2_plan.md`
(calibrated LLM-judge).

---

## 1. What v0.3 is

A single, defensible number — judge↔human agreement on a hand-labeled
calibration set — published in `docs/calibration.md` alongside per-category
breakdowns and a confusion matrix. Plus a working `LLMJudge` evaluator
class wired into the v0.2 harness, plus `pass^k` consistency support on
demand (local only, not in CI).

v0.1 was structure. v0.2 was rigor. v0.3 is the measurement that proves
the rigor caught something real. The calibration number is the only
artifact that matters; everything else in this release is scaffolding
to make that number defensible.

---

## 2. What v0.3 is explicitly NOT

- **A behavioral-realism gold set scored by the judge.** Deferred to
  v0.4. Building a judge-scored realism set inside the release that
  measures whether the judge is trustworthy is circular. Wait for
  v0.3's number before deciding whether the judge has earned a realism
  set.
- **Prompt tuning on FM-2.6 / FM-3.1.** The prompts shipped in v0.1.1
  are what we measure. If v0.3's number comes back low, prompt
  iteration is v0.4 work — not bundled into the release that's
  measuring the existing prompts.
- **PyPI publish.** Still gated on external usage signal.
- **A dashboard or web UI.** Still cut; CI gate output is the
  dashboard.
- **CI integration of judge-scored cases.** The `eval-gate` workflow
  stays deterministic-only. Judge flakiness is not allowed to break
  the merge gate.

---

## 3. How v0.3 will be judged

A v0.3 release is defensible if and only if all of the following hold:

1. **All 20 labels are by the maintainer's hand**, not by Claude or any
   other LLM. The agent drafts scenarios (reasoning + action pairs,
   task + output pairs); the human assigns the expected verdict for
   each.
2. **Labels are frozen before the judge runs.** Practical protocol:
   the maintainer commits the labeled JSONL in a dedicated commit
   ("freeze calibration labels"). The calibration script is first
   executed only after that commit lands. No retroactive label
   editing once the judge has been seen.
3. **The calibration report includes all four metric families:**
   raw accuracy, precision/recall/F1 per class, Cohen's kappa, and
   per-category breakdown. Kappa is the headline; the others go
   alongside for readability and for what an interviewer will
   actually probe.
4. **Per-category numbers isolate the audit's named failure modes.**
   The categories from `docs/judge_audit.md` (multi-step-plan
   false-positive, semantic-drift false-negative, truncation
   interaction for FM-2.6; confident-tone-incomplete, terse-correct,
   truncation interaction for FM-3.1) must each have at least one
   case in the set and a separate row in the per-category table.
5. **The methodology section in `docs/calibration.md` is rigorous
   enough that an interviewer cannot pick it apart.** It states
   explicitly: how cases were drafted (Claude), how labels were
   assigned (single human, before judge output was seen), why N=20
   not 50 (time-bounded scope; trade-off acknowledged), what
   "expected verdict" means for each case, and the known limitations
   (small N, single labeler, no inter-annotator agreement, prompts
   not tuned).
6. **The headline kappa, wherever it appears in this repo (the
   `README.md`, release notes, `docs/calibration.md` summary, any
   future blog post or social card committed to the repo), is
   reported with N and labeler-count inline.** Format:
   `κ = 0.XX (N=20, single labeler)`. Never as a standalone
   percentage. The headline number must not be allowed to travel
   without its caveat — screenshots, tweets, and slides quote the
   first appearance, and the methodology section is no protection
   if the number ships bare.

If any of these is missing, the calibration number is theatre. If all
are present, it is a defensible artifact and the release ships.

## 4. Phase A — LLMJudge evaluator + wiring tests

### Scope

Build `processguard/evaluators/llm_judge.py`. One parameterised
`AssertJudgeVerdict(detector="FM-2.6", expected="mismatch")` evaluator
that:

1. Constructs the events the detector expects (REASONING + TOOL_CALL
   for RAM; MESSAGE + TERMINATE for PT).
2. Invokes the detector's real `_judge()` method.
3. Compares the verdict's `failure_name` (or absence) against
   `expected`.
4. Records the verdict, confidence, and raw judge response text into
   the assertion result's `evidence` field for inspection later.

Mocked unit tests in CI — no live Anthropic calls in `pytest`. One
manual smoke test (a single live judge call costing ~$0.001) is part
of Phase A's success criterion, run on the maintainer's machine, not
in CI.

### Files

- New: `processguard/evaluators/llm_judge.py`
- Modify: `processguard/evaluators/registry.py` (register the new
  evaluator class)
- Modify: `processguard/evaluators/__init__.py` (re-export)
- New: `tests/test_llm_judge.py` — mocked LLM responses, covers the
  match/mismatch parsing path, confidence-floor handling, and the
  empty-output bypass path for FM-3.1.

### Success criterion

`pytest tests/test_llm_judge.py -q` passes with zero network calls.
Manual smoke (one real Anthropic call against a hand-crafted MISMATCH
case) returns the expected verdict and `evidence["judge_raw"]` matches
the model's actual output.

### Open questions

- Two evaluator subclasses (`AssertRAMVerdict`, `AssertPTVerdict`) or
  one parameterised class? I'd go parameterised. One class, two
  detector-mode constants. Less duplication.
- How does the evaluator's case definition surface the FM-3.1
  empty-output bypass path? Probably as a `category: empty_output`
  tag in the case JSONL, and the evaluator skips the live LLM call
  for those cases (matches the detector's own bypass logic).

### Anthropic spend

CI: $0. Mock-only.
Local: ~$0.001 (one smoke call).
Phase A total budget: rounding error.

## 5. Phase B — Calibration set + script + doc + `pass^k`

### Scope

Four sub-deliverables, all landing in one PR after the
freeze-labels commit lands first:

**5.1 Generate 30 candidate scenarios (15 per detector).** Claude
drafts:

- 15 RAM scenarios: 5 multi-step-plan, 5 semantic-drift, 3 truncation,
  2 clean-match controls.
- 15 PT scenarios: 5 confident-tone-incomplete, 4 terse-correct, 3
  truncation, 3 clean-complete controls.

Output: `datasets/calibration/ram_candidates_v1.jsonl` and
`pt_candidates_v1.jsonl`. Each line: `{id, category, reasoning_or_task,
action_or_output, expected_verdict: null}`. The `expected_verdict`
field is deliberately null at draft time.

**5.2 Maintainer picks 10 of each and labels them.** From the 30
candidates, the maintainer (you, by hand) picks the 10 strongest
cases per detector and fills the `expected_verdict` field for each.
Frozen as a single git commit titled
`freeze calibration labels (v0.3, N=20)`. The dropped cases stay in
`*_candidates_v1.jsonl` for reference (so the per-category coverage
choices are auditable). The committed set is
`datasets/calibration/ram_v1.jsonl` and `pt_v1.jsonl`.

**5.3 Calibration script.** `scripts/run_calibration.py`. Loads
labeled JSONL, runs the judge on each case, computes:

- Confusion matrix (2×2 per detector).
- Raw accuracy (% correct).
- Precision / recall / F1 per class.
- Cohen's kappa (corrected for chance agreement).
- Per-category accuracy breakdown.
- Cost summary in dollars (token counts × Haiku rates).

Writes:
- `calibration_results/v0.3_{detector}.json` — full per-case results.
- `docs/calibration.md` — the public artifact, with the headline
  number + per-category table + methodology section.

**5.4 `pass^k` support in the harness.** Add `--repeat k` flag to
`scripts/run_eval.py`. When `k > 1`:

- Each case runs k times under the same configuration.
- Report adds a `pass^k` column showing the fraction (e.g. "4/5"), not
  a boolean. Flaky-case identification is the whole value.
- Default k=1 unchanged — CI does not multiply API spend.

### Files

- New: `datasets/calibration/ram_candidates_v1.jsonl` (15 scenarios,
  labels null)
- New: `datasets/calibration/pt_candidates_v1.jsonl` (15 scenarios,
  labels null)
- New: `datasets/calibration/ram_v1.jsonl` (10 scenarios, labels
  filled — committed by the maintainer in the freeze commit)
- New: `datasets/calibration/pt_v1.jsonl` (10 scenarios, labels
  filled — committed by the maintainer in the freeze commit)
- New: `scripts/run_calibration.py`
- New: `docs/calibration.md`
- Modify: `processguard/harness/runner.py` (add `repeat` parameter,
  return per-attempt results when k>1)
- Modify: `processguard/harness/report.py` (render `pass^k` column
  when k>1; suppress otherwise)
- Modify: `scripts/run_eval.py` (add `--repeat` flag, plumb to
  Harness)
- New: `tests/test_repeat.py` (verify k=1 unchanged, k=2 reports
  fraction correctly, k=N runs the case exactly N times)
- New: `tests/test_calibration_loader.py` (case format, label
  validation, category coverage check)

### Phase B execution order

1. Phase A merges first (gate passes; `LLMJudge` evaluator exists
   even if no calibration cases use it yet).
2. Claude opens a PR with the 30 candidates committed and a draft
   `scripts/run_calibration.py` (mocked LLM, runs against fake
   labels). Maintainer reviews scenario shape, merges if good.
3. Maintainer opens the freeze-labels PR locally: edits the 20 chosen
   cases in `*_v1.jsonl`, fills labels, commits with the exact
   title. No Claude involvement in this step. Merges through the gate.

   **Ambiguity-exclusion protocol.** If the maintainer finds a
   candidate ambiguous during labeling — neither verdict is clearly
   correct — the case is excluded rather than force-labeled. The
   excluded case is moved back to the candidates file with a
   `rejected_for_ambiguity: true` flag (kept for audit). The
   next-best candidate from the 15-per-detector draft is promoted
   in its place. **The 20-case set is final composition, not final
   size:** if more than 5 candidates per detector get rejected for
   ambiguity, the protocol calls for Claude to draft a v2 batch
   rather than padding the set with forced labels. Forced labels
   poison the calibration more than excluded cases would.

   **Recommended labeling cadence:** one sitting (3-4 hours) or
   two consecutive sittings within 24 hours, to reduce intra-labeler
   drift across the set. Splitting labeling across many days lets
   the maintainer's interpretation of "MISMATCH" or "INCOMPLETE"
   shift between sessions, which corrupts the calibration in a way
   that doesn't show up in any single metric. Block out an evening
   with energy for ambiguous-case judgment; do not label tired.
4. Claude (or maintainer) runs `scripts/run_calibration.py` against
   the frozen labels with `ANTHROPIC_API_KEY` set. The script writes
   `calibration_results/` and updates `docs/calibration.md`. Final PR
   bundles the results, the `--repeat k` harness changes, and the
   docs update.

This sequencing is the load-bearing part of the protocol. Skipping
step 3 (or compressing it into a "Claude drafts labels, maintainer
spot-checks") collapses the entire credibility of v0.3.

### Success criterion

- `ram_v1.jsonl` and `pt_v1.jsonl` exist on main, each with 10 cases,
  every `expected_verdict` non-null, frozen in a commit titled
  `freeze calibration labels (v0.3, N=20)`.
- `scripts/run_calibration.py --detector ram` produces the JSON
  report and prints the markdown table. Same for `--detector pt`
  and `--all`.
- `docs/calibration.md` exists on main with: headline kappa numbers,
  raw accuracy alongside, per-category table, methodology section
  meeting the section-3 criteria above.
- `python scripts/run_eval.py --gold datasets/gold/v0.2.jsonl
  --repeat 3` runs each gold case three times and reports `pass^3`
  as fractions. Default behaviour without `--repeat` is unchanged.

### Anthropic spend

- 20 cases × ~150 input + ~80 output tokens × Haiku pricing ≈ $0.05
  per full calibration run.
- Expect 2-3 runs across iteration (catch a bug in the script,
  re-run; tweak a fixture, re-run). Total Phase B: $0.10–$0.20.
- `--repeat k=3` on the 4 LLM-gated v0.2 gold cases × ~150 tokens
  each ≈ $0.003 per `--repeat` invocation. Run a couple times to
  test. Total: rounding error.
- **Set a $5/month spend cap on the Anthropic key as the actual
  safety net.** Don't rely on per-script cost estimation alone.

### Open questions

- Should the labeling tool be a tiny CLI (`python scripts/label_case.py
  ...`) or just hand-edit JSONL? I'd say hand-edit — 20 cases is
  small enough that a CLI is overkill.
- The FM-3.1 PT detector has an explicit-bypass path (lines 73-85)
  for empty outputs. Should the calibration set include cases that
  exercise this path? My take: no. It's a deterministic path, not
  a judge path. Excluded from calibration; documented as such in
  methodology.

## 7. What v0.4 will look like (sketched, not committed)

For continuity — v0.3's design choices should leave room for these:

- **Behavioral realism gold set** scored by the now-calibrated judge.
  Lives in `datasets/gold/v0.4_realism.jsonl`. Run via
  `scripts/run_realism.py` (manual cadence, cost preview, no CI).
  Results go in `docs/realism_results.md` updated by hand on a
  quarterly-ish basis.
- **Prompt tuning** on FM-2.6 / FM-3.1 if v0.3's number warranted it.
  Tracked via the calibration set as a regression: any prompt change
  must improve kappa without dropping any per-category number.
- **Inter-annotator agreement**: second labeler on the same 20 v0.3
  cases. Cohen's kappa between humans gives a ceiling for what the
  judge could realistically hit. The current "single labeler" gap
  in v0.3's methodology becomes a v0.4 closure.

None of this is committed scope. Listed only so v0.3's design
doesn't paint v0.4 into a corner.

## 8. Execution rules (carry-over from v0.1.1 / v0.2)

- When given a specific implementation directive, implement it. Do
  not propose alternative approaches. Ask one clarifying question
  if ambiguous, then execute.
- When a test fails, do not retry the same fix with a small variation.
  Stop, summarise, propose at most two genuinely different
  alternatives, wait for the user's pick.
- "Tests pass" is a verified claim. Run the tests, read the output,
  then say done.
- Do not modify files outside the scope of the current phase.
- Status post between phases: "Phase X complete. Findings: [one
  sentence]. Proceeding to phase Y."
- 30-minute stuck rule: if no progress on the current phase for 30
  minutes, stop and report.

One v0.3-specific rule: **Claude does not write or edit labels, and
does not adjudicate label-judge disagreements after the fact.**

The 20 labels in `ram_v1.jsonl` and `pt_v1.jsonl` are the maintainer's
work product. If Claude is asked to "just suggest labels to save
time", that request must be declined; the methodology section
depends on the labels being unmediated. Drafting scenarios is fine.
Drafting labels is not.

The harder version of this rule applies after the calibration runs:
if the judge gets a verdict that disagrees with the maintainer's
label, there is a strong temptation to ask Claude "does my label or
the judge's verdict seem more correct?" That is label-editing in
disguise — the agent's framing of the case becomes the deciding
voice, exactly the dependency the methodology was designed to avoid.

If the maintainer, after reviewing a disagreement on their own,
genuinely wants to flip a label, the flip happens by hand in a
dedicated commit titled `relabel calibration case <id> (v0.3,
post-judge-review)`, and the calibration is re-run from scratch.
The total flip count is reported in `docs/calibration.md` as a
separate line in the methodology section — a high flip count is
itself a finding (labels weren't decisive enough) and shouldn't be
hidden.
