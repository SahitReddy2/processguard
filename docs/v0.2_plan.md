# ProcessGuard v0.2 — plan

**Status:** approved 2026-05-20, execution in progress.
**Supersedes:** the `ProcessGuard_v0.2_Build_Spec.md` draft from before v0.1.1
shipped. That draft has factual drift from the actual codebase (detector
names, framework adapters, infra claims) and is no longer the source of
truth; this doc is.

---

## 1. What v0.2 is and isn't

**Is:** an offline evaluation harness + CI gate. The first version of
ProcessGuard where every change can be locked in against a fixed set of
regression cases.

**Isn't:**
- A calibrated LLM-as-judge (Phase 2 in the spec — deferred to v0.3).
- A `pass^k` consistency reporter (Phase 3 — deferred to v0.3).
- An online evaluation sampler (Phase 4 — deferred to v0.3 or later).
- A dashboard, web UI, or PyPI release (Phase 6 — explicitly cut from
  v0.2 and likely deferred indefinitely; the CI gate's PR-comment output
  *is* the v0.2 dashboard).
- A working CrewAI adapter (kept in `experimental/`, targeted v1.1).

The cut is deliberate. The spec's "if you only have 2 weeks: Phase 0 → 1
→ 2 → 5 + minimal dashboard" was already tight. v0.2 takes the more
honest cut: Phase 0 + 1 + 5 only, with **zero LLM API key requirements**
for the in-scope deliverable. Phase 2's LLM-judge calibration becomes
v0.3 because it (a) needs a sustained API budget, (b) needs hand-labelling
that is its own focused task, and (c) builds on Phase 1 anyway.

---

## 2. Reconciliation against the build spec

The spec's *thesis* is correct: evaluation is the 2026 bottleneck, three
evaluator types used together is the right shape, `pass^k` beats
`pass@k`, and a CI gate is what distinguishes this from every other
AI-tooling repo. Adopted as-is.

The spec's *factual claims about v0.1* are wrong. Corrections (already
verified against the v0.1.1 codebase):

| Spec said | Actual |
|-----------|--------|
| Detectors: deadlock, duplicate execution, context drift, goal conflict, escalation failure | FM-1.3 step repetition, FM-1.5 unaware termination, BEYOND-MAST no-progress loop, FM-2.6 reasoning-action mismatch, FM-3.1 premature termination |
| Pydantic event schemas | `@dataclass` — no Pydantic dependency |
| FastAPI endpoints | None |
| Docker packaging | No Dockerfile |
| CrewAI integration shipped | In `experimental/`, doesn't capture tool events on current CrewAI, deferred to v1.1 |

The spec's *Repo Structure* and *Tech Stack* sections both assume the
above infra exists. This plan does not. Anything we add (FastAPI,
Docker, Postgres) is added on demand, not for resume aesthetics.

### One lesson from v0.1.1 that reshapes the spec

v0.1.0's synthetic demos passed cleanly but the LangGraph adapter had
two silent bugs (double-trace, missing TERMINATE) that only surfaced on
the first real-agent run in Item 4. The eval harness must run against
real agent traces — including ones from real adapter integrations — as
part of its acceptance criteria. Otherwise we're back to "looks fine in
isolation, breaks in integration."

Concretely: Phase 1 does not ship until its gold set includes cases
that exercise both `synthetic_*.py` demos AND `real_langgraph_demo.py`,
even if the LLM cases SKIP in CI for lack of secrets.

## 3. Approved decisions (the five questions from the planning round + refinements)

1. **Sequencing**: no-LLM-first. Phases 0 + 1 + 5 ship without any API
   keys. Phase 2 (LLM-judge calibration) becomes v0.3.

2. **CrewAI**: stays deferred to v1.1. Not in v0.2 scope. Pulling the
   CrewAI rewrite into the eval-harness phase dilutes both deliverables.

3. **Dashboard**: cut entirely from v0.2. The CI gate's PR comment +
   uploaded JSON artefact *is* the dashboard. Polished web UIs are a
   v1.0 problem at the earliest, possibly never (Ragas doesn't have one;
   ruff doesn't have one).

4. **First acceptance criterion for Phase 1**: option (a) — runs against
   `examples/real_langgraph_demo.py` + the two `synthetic_*.py` demos.
   Do not build a multi-agent supervisor example speculatively.

5. **PyPI publish**: held for v1.0. `pip install git+...` is the install
   path through v0.2. Wait for an external user to ask before publishing.

**Refinements added during plan review:**

- **Unknown assertion types must raise a clear error.** Loader behaviour:
  `AssertionTypeNotRegistered: "FooBar" — registered: [AssertToolCalled,
  AssertWithinStepBudget, AssertDetectorFired]`. Saves an hour of
  debugging the first time someone fat-fingers a class name.
- **PR comment must surface the skip count visibly.** Format:
  `10/10 deterministic ✅ • 3/3 LLM cases SKIPPED (ANTHROPIC_API_KEY not
  set in CI)`. Not buried in the JSON artefact. The green badge must not
  overclaim.
- **`eval_results/` and `*.db` / `processguard.db` go in `.gitignore`
  pre-emptively.** Run artefacts are not source; accidental commits
  produce diff noise every CI run.
- **Workflow setup duplication is acceptable in v0.2.** If
  `tests.yml` and `eval-gate.yml` end up sharing a lot of setup
  (checkout, Python install, deps), refactor to a composite action.
  Not a blocker for v0.2 — flag it during Phase 5 and decide then.
- **Gold-set split**: regression-heavy in v0.2. Five of ten cases are
  regressions for the v0.1.1 bug discoveries. Behavioural-realism cases
  come in v0.3 as their own JSONL file (`datasets/gold/v0.3_realism.jsonl`),
  scored by the calibrated LLM-judge, kept separate so CI strictness
  doesn't depend on judge stability.

## 5. Phase 1 — Eval harness + deterministic evaluators + 10-case gold set

**Scope.** Three deliverables.

### 5.1 EvalCase schema

JSON object per line in JSONL files. Python class refs, no DSL.

```json
{
  "id": "regression-step-rep-001",
  "source": "v0.1.1 regression",
  "notes": "FM-1.3 must fire at 3 repeats, not 2",
  "agent_target": "manual",
  "input": {
    "events": [
      {"type": "tool_call", "tool_name": "web_search", "tool_args": {"query": "X"}},
      {"type": "tool_call", "tool_name": "web_search", "tool_args": {"query": "X"}},
      {"type": "tool_call", "tool_name": "web_search", "tool_args": {"query": "X"}}
    ]
  },
  "assertions": [
    {"type": "AssertDetectorFired", "args": {"failure_mode": "FM-1.3"}}
  ]
}
```

`agent_target` values:
- `"manual"` — `input.events` is a list of `AgentEvent` dicts to push
  through `guard.emit()` directly. No LLM needed.
- `"examples/<script>.py"` — run the example script with `input.task` as
  the user task (script must expose a `run_with_task(task)` callable).
  Counts as LLM-required if the script imports `langchain_anthropic` or
  `langchain_google_genai`.

### 5.2 Harness class

`Harness(cases: list[EvalCase])`. For each case:

1. Resolve assertions: instantiate each `assertions[i]["type"]` class
   from a registry; raise `AssertionTypeNotRegistered` with the full
   registered list if unknown.
2. Detect LLM-required cases by inspecting `agent_target`. If the
   required env var (`ANTHROPIC_API_KEY` / `GOOGLE_API_KEY`) is missing,
   mark the case `SKIPPED` with a reason; do NOT raise.
3. For each non-skipped case:
   - New `ProcessGuard` with per-case `:memory:` SQLite for isolation.
   - Run the agent / emit the manual events.
   - Apply each assertion against the collected events + detections.
   - Record `PASSED` / `FAILED` per assertion.
4. Aggregate into an `EvalReport` (dataclass): per-case status, per-assertion
   verdicts, summary counts, total runtime.

`SKIPPED` counts as build-passing. `FAILED` does not.

### 5.3 Three deterministic evaluators

- `AssertToolCalled(tool_name: str, min_times: int = 1)` — passes iff
  the agent emitted ≥ `min_times` `TOOL_CALL` events with matching
  `tool_name`.
- `AssertWithinStepBudget(max_events: int)` — passes iff total event
  count for the trace ≤ `max_events`.
- `AssertDetectorFired(failure_mode: str)` — passes iff
  `guard.policy.detections` contains a detection whose `failure_mode`
  matches.

More evaluators come later; these three cover the gold set.

### 5.4 10-case gold set (regression-heavy)

`datasets/gold/v0.2.jsonl`. Each case has a `source` field pointing to
where it came from.

| # | id | Source | LLM-req? |
|---|---|---|---|
| 1 | regression-double-trace | v0.1.1 Bug 1 | Yes (LangGraph) |
| 2 | regression-storage-thread | v0.1.1 Bug 1b | No |
| 3 | regression-terminate-emitted | v0.1.1 Bug 2a | Yes (LangGraph) |
| 4 | regression-step-rep-boundary | StepRepetition contract | No |
| 5 | regression-no-progress-fanout-allowed | NPL contract "must not fire" | No |
| 6 | regression-premature-empty-output | PT empty-bypass path | No |
| 7 | synthetic-raw-loop-step-rep | synthetic_raw_loop_demo | No |
| 8 | synthetic-raw-loop-no-progress | synthetic_raw_loop_demo | No |
| 9 | synthetic-langgraph-loop | synthetic_langgraph_demo | Yes (Anthropic) |
| 10 | real-langgraph-completion | real_langgraph_demo | Yes (Google) |

Cases 1, 3, 9, 10 SKIP in CI without secrets. The other 6 run unconditionally.
That gives a strong CI signal (6 deterministic regressions enforced) with
no API key dependency.

A `datasets/gold/README.md` documents each case's intent in one sentence.

### 5.5 Files

```
processguard/
  evaluators/
    __init__.py
    base.py            # Evaluator ABC + EvalResult dataclass
    registry.py        # registry + AssertionTypeNotRegistered
    deterministic.py   # AssertToolCalled, AssertWithinStepBudget, AssertDetectorFired
  harness/
    __init__.py
    eval_case.py       # EvalCase dataclass
    runner.py          # Harness class
    report.py          # EvalReport + markdown printer (used by both CLI and CI)
datasets/
  gold/
    v0.2.jsonl
    README.md
scripts/
  run_eval.py          # CLI: python scripts/run_eval.py --gold datasets/gold/v0.2.jsonl
tests/
  test_eval_harness.py
  test_evaluators.py
  test_eval_registry.py  # unknown-type error path
```

### 5.6 Success criterion

`python scripts/run_eval.py --gold datasets/gold/v0.2.jsonl` produces a
report. With no API keys set: 6/6 deterministic cases pass, 4/4 LLM
cases SKIP, exit code 0. With both keys set: 10/10 pass, exit code 0.
With a deliberately-broken detector: exit code non-zero, report shows
which case(s) failed and why.

Report writes to `eval_results/v0.2_{ISO-timestamp}.json` AND prints
the markdown table to stdout (the same table the CI gate will post).

`eval_results/` is gitignored.

## 6. Phase 5 — CI gate

**Scope.** Two GitHub Actions workflows.

### 6.1 `.github/workflows/tests.yml`

Runs on push and PR. Matrix on Python 3.10 / 3.11 / 3.12. Runs `pytest`.
Fails the build on any test failure.

### 6.2 `.github/workflows/eval-gate.yml`

Runs on PRs only (push-to-main triggers the same on the merge commit).
`paths-ignore: ['**/*.md', 'docs/**']`. Steps:

1. Checkout + Python setup (single version, 3.12).
2. `pip install -e .[langgraph,dev]`
3. `python scripts/run_eval.py --gold datasets/gold/v0.2.jsonl --output eval_results/ci.json --format markdown` (capture markdown to a file).
4. If exit code non-zero, fail the build but still post the comment.
5. Post the markdown report as a PR comment via
   `peter-evans/create-or-update-comment@v4` using the default
   `GITHUB_TOKEN`. Comment must be edited-in-place on subsequent runs
   (same comment ID), not appended.
6. Upload `eval_results/ci.json` as a workflow artefact.

**Comment header format**:
```
**ProcessGuard eval gate** — 6/6 deterministic ✅ • 4/4 LLM cases SKIPPED (no API keys in CI)
```

The SKIPPED count must be on the first line. The build-status badge
cannot overclaim.

### 6.3 Branch protection

Documented in `CONTRIBUTING.md` only:

> The `main` branch should be configured (in repo settings → Branches)
> to require both `tests` and `eval-gate` workflows to pass before merge.
> This is a manual repo-settings step, not enforced by the workflow files
> themselves.

Not enabled via API. User toggles when external contributors arrive.

### 6.4 Files

- New `.github/workflows/tests.yml`
- New `.github/workflows/eval-gate.yml`
- Modify `README.md` — replace static "tests passing" placeholder with
  live workflow badge.
- Modify `CONTRIBUTING.md` — add branch protection note + how to run
  the eval locally.
- Modify `.gitignore` — add `eval_results/`, `*.db`, `processguard.db`
  (if not already covered by the existing `*.db` line — actually it is;
  just verify and document).

### 6.5 Success criterion

A no-op PR triggers both workflows, both go green, eval-gate posts a
comment showing 6/6 deterministic + 4/4 skipped. A PR that deliberately
breaks a detector (comment out a `Detection(...)` return) turns
eval-gate red and the comment names the failing case(s).

## 8. What v0.3 will look like (sketched, not committed)

For continuity — Phase 1's design choices should be forward-compatible
with these:

- **Phase 2 — calibrated LLM-judge.** Adds `processguard/evaluators/llm_judge.py`
  + `datasets/gold/v0.3_realism.jsonl` (behavioural cases scored by the
  judge). Reuses Phase 1's `Harness` and `EvalReport`. The judge's
  precision/recall on a hand-labelled calibration set becomes the headline
  metric for v0.3.
- **Phase 3 — `pass^k`.** `Harness` gains a `--repeat k` flag, each case
  runs k times, the report adds a `pass^k` column.
- **Phase 4 — online evaluation.** Async trace sampling hook + reference-free
  evaluators. Out of v0.3 scope likely.

None of this is committed scope. Listed only so the v0.2 design doesn't
paint v0.3 into a corner.

## 9. Execution rules (carry-over from v0.1.1)

Same working rules as the v0.1.1 direction-correction pass:

- When given a specific implementation directive, implement it. Do not
  propose alternative approaches. Ask one clarifying question if
  ambiguous, then execute.
- When a test fails, do not retry the same fix with a small variation.
  Stop, summarise, propose at most two genuinely different alternatives,
  wait for the user's pick.
- "Tests pass" is a verified claim, not an assumption. Run the test,
  read the output, then say done.
- Do not modify files outside the scope of the current phase. Out-of-scope
  changes get flagged separately for review.
- Status post between phases: "Phase N complete. Findings: [one sentence].
  Proceeding to phase N+1."
- 30-minute stuck rule: if no progress on the current phase for 30 min,
  stop and report.

## Post-ship updates (2026-06-07)

The plan above is preserved as the design contract. The following
landed after v0.2.0 shipped and are recorded here so the doc keeps
pace with reality.

### Branch protection — enabled

Branch protection on `main` is now active (configured via the GitHub
API on 2026-06-07, not just documented as a "manual step the user
should do"):

- Required status checks: `pytest (3.10)`, `pytest (3.11)`,
  `pytest (3.12)`, `run-eval`. Strict mode (PRs must be up to date
  with main before merge).
- `enforce_admins: false` — admin can bypass in emergencies; solo-dev
  safety valve.
- `required_linear_history: true` — matches the squash-merge style.
- `allow_force_pushes: false`, `allow_deletions: false`.
- `required_pull_request_reviews: null` — solo project; add when
  contributors arrive.

The CONTRIBUTING.md note still says "configure in repo settings" since
that remains the right path for future maintainers; the current
configuration is recorded above for reference.

### Eval-gate workflow — `paths-ignore` bug fixed

The original `eval-gate.yml` used `paths-ignore` to skip docs-only
PRs. With branch protection requiring the `run-eval` check, this
created a permanent BLOCKED state: GitHub treats a required check
that never runs as "pending forever." Discovered on PR #2 (the docs
update closing the v0.1.1 residual); fixed in PR #3.

New behaviour: the workflow always triggers on PRs. The first step
diffs changed files against the PR base. If everything is `*.md`,
`docs/`, or `.github/ISSUE_TEMPLATE/`, the workflow prints a skip
notice and exits success in ~8s without installing dependencies or
running the eval. If any code or workflow file changed, the full
eval runs as before.

### v0.2.1 — housekeeping patch release

Four small fixes from the v0.2.2 probe and from running the gate
end-to-end:

1. **`duckduckgo_search` → `ddgs`.** The old package emits a
   deprecation warning on every call. Demo now imports `ddgs` first,
   falls back to the old name for users with un-migrated environments.
2. **DDG empty-result fallback.** v0.1.1's in-memory KB fallback only
   triggered on `Exception`; the v0.2.2 probe showed DDG returning
   `(no results)` for three reasonable queries without ever raising.
   Now after 2 consecutive empty results, the fallback engages for
   the rest of the run.
3. **`ProcessGuard.close()`** + context-manager support + safe
   `__del__`. Closes the SQLite connection explicitly. Eliminates the
   `ResourceWarning: unclosed database` that fired on guard GC.
   5 new regression tests in `tests/test_close.py`.
4. **This doc updated** to reflect post-ship reality.

Test count: 59 → 64. All deterministic gate cases still pass.

### What v0.2.1 did NOT do

- Did not bump scope into the v0.3 LLM-judge calibration work — that
  is its own focused effort and stays out of patch releases.
- Did not migrate the synthetic demos. Their behaviour is unchanged.
- Did not change branch protection requirements or workflow paths
  beyond the bug fix in PR #3.
